samograph.dev — Product Brief

Zero-setup hosted samograph: paste a meeting link, get a live shareable transcript — then hand your AI agent a secure link to listen and act.

SPEC v0.4 · 2026-06-24 · post-publish stakeholder-feedback round · authored with samospec · 2 prior Claude review rounds applied · human-skimmable summary of the authoritative SPEC.md

v1 — MVP, build this week v2 — designed-for, deferred open question / risk

01Goal & why

Goal. A hosted, fully-managed SaaS at samograph.dev that wraps the existing samograph CLI and its Recall.ai meeting bot so an end user never runs a CLI, holds a Recall token, or operates a tunnel. v1 delivers exactly one experience: sign in → paste a Zoom or Google Meet URL → the bot is in the call → a per-call page streams a live, persisted transcript that's also shareable read-only. v2 layers a secure bidirectional AI-agent channel on top of any active call.

Why it exists

  • samograph is fundamentally about AI agents participating in live meetings (Claude Code, Codex, …). The CLI proves the surface — join, watch, chat, presence, frame, frames, leave — but forces every user to install Bun, hold a RECALL_API_KEY, run a local callback server, and operate an ngrok/cloudflared tunnel.
  • That setup tax is the only thing between "AI agent in your meeting" and a turnkey product. v1 removes the tax; v2 mounts the scoped, revocable AI-agent channel on top.
  • v1 alone (zero-install, live, persisted, shareable transcript) already differentiates against Otter / Fireflies / Read / tl;dv for this use case. v2 is the strategic differentiator no general meeting-notes product offers.

Explicit v1 non-goals (do not silently re-introduce)

No calendar integration · no Google OAuth · no AI features in the product UI · no billing / plans / seats · no video storage · no AI-generated summaries · no email digests · no MS Teams · no branded bot identity · no multi-language UI. The Phase-2 AI-agent channel is designed for but not built in v1 — only the seams ship.

02Scope phases

v1 — MVP (this week)

Zero-setup live transcript

Marketing + app at samograph.dev. Email magic-link auth only. One primary action: paste a Zoom / Meet URL → bot joins via a shared server-side Recall key → per-call page streams the transcript live over WebSocket ([timestamp] Speaker: utterance) → shareable read-only via a signed link → persisted to Postgres in real time and stored indefinitely. Plus: transcription dictionary + language settings (Deepgram), and GDPR data deletion (per-call + account, incl. the Recall recording).

v2 — Phase 2 (pre-wired)

AI channel · billing · isolation

Secure AI-agent channel for an active call: owner mints a per-call, capability-scoped, short-TTL, revocable token. LISTEN = live transcript + backfill. ACT = remote chat/frame/frames/presence/leave. HTTP+WS API + MCP endpoint, hard isolation, full audit, rate limits, one-click revoke. v2 also adds Stripe per-seat billing, video storage (R2/B2), expanded settings, and VM-grade tenant isolation.

v3+ — explicitly deferred

Not in v1/v2's design

Branded/consistent bot identities; post-call transcript email; multi-language UI; synced transcript+video viewer (click-a-word-to-seek); MS Teams + Webex; native integrations (Slack/Notion/CRM/Zapier); public REST API; EU data residency; self-hosted Firecracker on Hetzner bare-metal as a cost optimization over managed microVMs.

v1 ships the seams for v2

A capability-token model (read/share now; act:* later — same verifier) · a loopback bot-worker command/act API mapping 1:1 to CLI verbs · a single authorizeCall tenancy gate · an audit_log table. The multi-region tunnel pattern is also proved in build-week (Sprint 3 second region) but a single region is sufficient to ship — multi-region is not a launch gate.

03Architecture at a glance

Browser / per-call page → app-api (magic-link auth, create call, mint share / agent tokens) → bot-orchestrator (holds the ONE shared Recall key) + token-service → Recall.ai → one persistent regional cloudflared named tunnel (not per-call ngrok) → ingest (webhook routed by bot_id, signature + ingest_secret verified, normalize, persist) → ws-hub fan-out → per-call page. A bot-worker exposes the chat/frame/presence/leave act API; Postgres is single multi-tenant with RLS. Transcription runs on Deepgram (via Recall).

Single v1 success metric — W1 activation ≥ 0.5. Fraction of new signups who, within their first week, (a) paste a meeting link, (b) get the bot admitted into a real call (Recall reaches in_call_recording, not first transcript line), and (c) watch ≥30 s of live transcript on the per-call page. Pickup latency (event received → status visible, p95 ≤ 1 s) is a secondary health metric only.

05v1 user stories

Each ships with a persona, outcome, and manual-test recipe in the spec (§3).

  • S1Zero-setup live transcript (primary JTBD): paste URL → per-call page opens ≤2 s, status JOINING ≤5 s of submit, IN_CALL on Recall in_call_recording; lines stream live; reopening the tab resumes without loss.
  • S2Share the live transcript read-only via a signed link — recipient reads live without signing in; revoke takes effect ≤1 s; share caps: ≤200 concurrent conns, ≤20 cmds/min/conn, ≤1000 establishments/hour/token.
  • S3Durable transcript after Recall's ~7-day video TTL — full final transcript with timestamps + speakers, downloadable as plain text identical to the CLI format. (Transcripts are stored indefinitely from day one.)
  • S4Bot fails to join → terminal COULD_NOT_JOIN with the Recall reason in plain English; "Try again" returns to the dashboard with the URL pre-filled (no implicit retry, one user action = one Call row).
  • S5Mid-call tunnel/ingest outage is loud, never silent — banner + a SAMOGRAPH-WARNING: tunnel unreachable line appended to the stream; recovery clears both. Driven by a separate ingest_degraded overlay (a call can be IN_CALL + degraded at once).
  • S6In-call recording disclosure — bot name samograph (recording) + a single chat post on in_call_recording. If Recall reports in_call_not_recording, no disclosure is posted; the call goes terminal COULD_NOT_RECORD and the bot leaves cleanly.

v2 stories (specified now to keep v1 architecture honest; not built in v1)

  • S7Mint a scoped AI-agent link for an active call — pick scopes (listen, act:chat, act:frame, act:presence, act:leave), TTL bounded by call lifetime; one-time-display token + URL; "Revoke" kills it immediately.
  • S8AI agent acts through the channel — subscribes to the live transcript, posts chat (same chime as the CLI), pulls a frame, sets presence. Never sees the Recall token or other tenants; rate-limited per token; every action audited.

06How v1 stays safe & honest

  • Shared Recall key, isolation above it. The key lives only in bot-orchestrator + ingest; every request that names a bot_id passes the single authorizeCall tenancy gate before any Recall call; the audit log captures every Recall-mediated action.
  • Webhook authenticity. Each /webhook POST is checked in order: Recall HMAC signature → known bot_id → constant-time ingest_secret match → tenancy gate. The ingest_secret is server-side-only, never a user-visible token.
  • Persisted vs derived scopes. read is derived from the owner's session (no tokens row); only share (v1) and act:* (v2) are persisted capability tokens — HMAC-SHA256, KID-rotated (90 d / 30 d overlap), jti-unique, constant-time verified, no verifier cache in v1.
  • WS fan-out + backpressure. Per-call_id channel, bounded queue (256 msgs OR 512 KB, drop-oldest + gap frame); a slow client never blocks others; publisher p99 ≤ 5 ms holds for k ≤ 32 stalled subscribers.
  • Loud, leader-elected watchdog. Per-region probe every 20 s; a Postgres advisory lock makes warnings fire exactly once per outage across ingest replicas.
  • Postgres RLS everywhere. Every tenant_id-bearing table filtered by tenant_id, set only by the tenancy gate (via transaction-local set_config).
  • Typed errors, never a silent hang. Every failure is a stable SAMO-… code with plain-English copy and defined client behavior (§5.16); call-path failures map to a terminal calls.status.
Engineering detail — RLS InitPlan fix (§5.10). Policies use tenant_id = (SELECT current_setting('app.tenant_id'))::uuid. The scalar sub-SELECT wrapper makes Postgres cache the predicate as a once-per-statement InitPlan instead of re-evaluating current_setting() per row (which would also defeat indexes) — mandatory on every policy, catastrophic on large transcript scans otherwise.

The subtle pieces are TDD-built: transcript normalizer, capability token verifier, WS backpressure, tenancy gate, magic-link security, webhook authenticity, bot-lifecycle/pickup-latency, worker discovery, share-scope limits (§6.2).

07Tenant isolation roadmap

Isolation has two layers; the spec is explicit about which one each phase buys (§4.8). This directly answers the "we want separate VMs / microVMs" concern as a phased roadmap, not a v1 gap pretending to be a VM.

  • v1 — data isolation (shipped). Single multi-tenant Postgres with Row-Level Security. Strong for data; says nothing about compute.
  • v1 — compute isolation (weaker, honest). Per-call bot-workers run as hardened containers (seccomp + user namespaces + AppArmor/SELinux, read-only rootfs, no host mounts). A container shares the host kernel — NOT a security boundary equal to a VM. Adequate in v1 because workers run our code, not tenant-supplied code.
  • v2 — VM-grade isolation. Move the bot-worker tier to managed Firecracker (Fly.io Machines) — rent the microVM boundary (networking solved, EU regions, scale-to-zero, ~$2/mo for an always-on 256 MB machine) rather than build a Firecracker fleet. gVisor is the fallback (no KVM needed, ~10–30% I/O overhead) if the worker tier must stay on Hetzner Cloud. Stateless services (app-api, ingest, ws-hub) stay on Hetzner Cloud.
  • v3 — cost optimization. Revisit self-hosted Firecracker on Hetzner bare-metal once volume justifies the ops (cheaper per unit, hardest ops).
Constraint that shapes the path. Hetzner Cloud (CX/CPX/CCX) has no nested virtualization — /dev/kvm is unavailable in the guest, so Firecracker / Cloud Hypervisor / Kata cannot run on Hetzner Cloud VMs. Only Hetzner dedicated / bare-metal exposes KVM. That is why the v2 microVM boundary uses a managed provider (or bare-metal in v3), not Hetzner Cloud.

Retention (§5.13)

  • Transcripts: indefinite from v1 — the durable product record (~50 KB / meeting-hour; cheap in Postgres well past 100 GB).
  • Video: now a v2 feature — v1 stores no video (relies on Recall's ~7-day window). v2 egresses recordings to R2 / B2 with configurable N-month retention (default 6 months) + daily auto-purge; video dominates storage cost (~0.5–1 GB/hr), so retention is bounded and Team-plan-gated.

Data deletion (§5.14 — GDPR baseline, shipped v1)

Per-call: removes transcripts + any stored video, revokes its tokens, and calls the Recall API to delete the Recall-side recording; keeps an audit tombstone. Account / all tenant data: purges everything, revokes sessions, deletes Recall recordings, honors a ≤30-day erasure SLA (active calls force-left first). Export-before-delete is offered in the flow.

09Billing (v2) & error codes

Billing & plans (§5.15 — v2, Stripe)

v1 is free behind cost guardrails. v2 introduces Stripe per-seat subscriptions matching Circleback's model (per-seat, unlimited meetings under fair-use; monthly + annual; trial; no permanent free tier).

Plan~Price / seat/mo (annual / monthly)Includes

Team~$25 / ~$30+ shared meeting library, retention controls + video storage, access controls, usage dashboard, central billing
Enterprisecustom+ SSO, advanced security, BAA, EU residency
Margin guardrail. Recall bills per-minute but per-seat pricing is flat, so heavy users can be unprofitable. Enforce fair-use minute caps per seat/month (friendly soft limit, not a mid-call cutoff), monitor per-tenant margin, and keep the v1 cost guardrails as the floor. This is the central billing risk (R-billing).
